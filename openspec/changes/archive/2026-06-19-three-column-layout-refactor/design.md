## Context

Wind 是一个基于 Tauri 2.0 的 Windows 文件工作站，当前采用传统的文件树+编辑器+终端三面板布局。用户反馈这种布局不利于文件目录间的快速跳转和预览，希望参考 Yazi 文件管理器进行重构。

**当前状态：**
- 布局：左文件树 + 右上编辑器 + 右下终端
- 编辑器：使用 textarea，功能有限
- 终端：固定在底部，占用空间

**约束条件：**
- 必须保持 vim 驱动的交互风格
- 需要支持 Windows 特定功能（ConPTY 终端）
- 现有预览系统（Shiki 语法高亮）需要保留

## Goals / Non-Goals

**Goals:**
- 实现三列文件浏览器布局（父目录、当前目录、预览/编辑器）
- 支持目录联动导航（参考 Yazi）
- 提供完整代码编辑功能（CodeMirror 6）
- 实现浮动终端（默认隐藏）
- 保持 vim 风格的快捷键操作

**Non-Goals:**
- 不支持多标签页（本次重构）
- 不支持远程文件系统
- 不实现文件搜索功能（已有 Ctrl+P）
- 不支持自定义主题（使用 one-dark）

## Decisions

### 1. 布局方案：CSS Grid vs Flexbox

**选择：CSS Grid**

理由：
- 三列等分/比例分配更直观
- 支持拖拽调整比例
- 响应式布局更容易实现

备选方案：
- Flexbox：需要计算 flex-basis，比例调整复杂

### 2. 编辑器选型：CodeMirror 6 vs Monaco vs textarea

**选择：CodeMirror 6**

理由：
- 轻量级（~100KB gzipped），适合桌面应用
- 模块化设计，按需加载语言包
- 良好的 Svelte 集成
- 支持 vim 模式（可选）

备选方案：
- Monaco：功能强大但体积大（~2MB），适合 IDE
- textarea：简单但无语法高亮

### 3. 状态管理：Svelte stores vs Context API

**选择：Svelte stores + 组件状态**

理由：
- 现有项目已使用 stores
- 跨组件状态共享简单
- 响应式更新自动处理

备选方案：
- Context API：适合深层嵌套，但本项目层级不深

### 4. 终端浮动方案：CSS position vs Portal

**选择：CSS position: fixed**

理由：
- 实现简单，不需要 Portal 机制
- 与现有浮层（命令面板）风格一致
- 性能好，不需要额外 DOM 操作

备选方案：
- Portal：适合跨层级渲染，但增加复杂度

### 5. 全屏编辑器：浮层 vs 路由切换

**选择：浮层（Overlay）**

理由：
- 保持三列布局状态
- 打开/关闭动画简单
- 与现有命令面板风格一致

备选方案：
- 路由切换：需要保存/恢复布局状态

## Risks / Trade-offs

### 风险 1：CodeMirror 6 集成复杂度
**风险**：CodeMirror 6 的模块化设计可能增加集成难度
**缓解**：先实现基础功能，后续逐步添加语言支持

### 风险 2：目录联动导航的性能
**风险**：频繁读取目录内容可能影响性能
**缓解**：
- 缓存已读取的目录内容
- 使用防抖处理快速导航
- 显示加载状态

### 风险 3：快捷键冲突
**风险**：新增快捷键可能与现有功能冲突
**缓解**：
- 使用 Ctrl+W 前缀避免冲突
- 提供快捷键配置选项（未来）

### 风险 4：浮动终端的焦点管理
**风险**：终端与其他面板的焦点切换可能混乱
**缓解**：
- 明确的焦点切换逻辑
- Escape 关闭终端并恢复焦点

### 权衡 1：功能完整性 vs 界面简洁
**权衡**：全屏编辑器提供完整功能，但增加了界面复杂度
**决策**：通过快捷键（e）触发，不干扰主界面

### 权衡 2：性能 vs 功能
**权衡**：CodeMirror 6 比 textarea 重，但提供更好体验
**决策**：按需加载，只在编辑时初始化
