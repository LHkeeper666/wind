## Context

DirectoryPanel 组件从后端读取目录内容并渲染文件列表。当前的文件列表完全由后端返回的 `FileEntry[]` 驱动，没有特殊的"虚拟条目"。用户需要通过父目录面板（左侧列）或键盘快捷键来返回上级目录。

项目采用 vim 风格的三列布局：父目录 | 当前目录 | 预览/编辑器。当前目录面板支持 `j/k` 导航、`Enter` 进入目录、`gg/G` 跳转首尾等快捷键。

## Goals / Non-Goals

**Goals:**
- 在当前目录面板文件列表顶部显示 `..` 条目
- `..` 条目支持与普通目录条目相同的交互：选中、Enter 导航、双击导航
- `..` 条目在 `gg` 跳转时作为第一个可选项（index 0）
- 根目录下不显示 `..` 条目

**Non-Goals:**
- 不改变父目录面板的行为
- 不添加新的后端命令
- 不改变现有的目录缓存策略

## Decisions

### 1. 在前端注入 `..` 条目而非后端返回

**选择**: 在 `loadDirectory` 获取文件列表后，前端在数组头部插入一个虚拟的 `FileEntry { name: '..', path: parentPath, is_dir: true }`。

**理由**: 后端 `read_directory` 返回的是文件系统真实条目，`..` 是 shell 约定而非真实目录项。在前端注入保持后端职责单一，且 parentPath 已在 layout store 中计算好。

**替代方案**: 修改 Rust 后端在 `read_directory` 中自动添加 `..` —— 侵入性更强，且影响所有调用方。

### 2. `..` 条目与普通条目统一处理

**选择**: `..` 条目作为 `FileEntry` 参与现有的选择、导航、滚动逻辑，不做特殊分支。

**理由**: 减少代码复杂度。`..` 的 `is_dir: true` 保证 Enter 时走 `onNavigate` 路径，与普通目录一致。唯一特殊处理是 `onNavigate` 需要传入父目录路径而非 `..` 条目的 path（虽然两者等价）。

### 3. 根目录检测

**选择**: 通过比较 `path` 和其父路径是否相同来判断是否为根目录。Windows 上 `C:\` 的父路径仍是 `C:\`。

**理由**: 跨平台兼容，不依赖硬编码的根目录判断。

## Risks / Trade-offs

- [缓存一致性] `..` 条目在缓存中存在，但 parentPath 可能因外部导航而变化 → 缓存命中时需要重新注入 `..` 条目以确保 parentPath 正确。缓解方案：在缓存命中后重新计算并注入 `..`。
- [selectedIndex 偏移] 注入 `..` 后，原有的 selectedIndex 逻辑（如 `gg` 跳到 index 0）自然将 `..` 作为首个条目，这是期望行为，无需额外处理。
